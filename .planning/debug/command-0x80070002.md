## Evidence

- 2026-05-12: `src-tauri/src/commands/shell.rs` 当前代码包含 wt.exe (Windows Terminal) 优先路径，spawn() 后如果 Ok 则直接返回，不走 cmd.exe 回退
- 2026-05-12: wt.exe 作为 UWP App Execution Alias，从 Tauri GUI 进程 spawn 时 spawn() 返回 Ok(()) 但 WT 内部报错 0x80070002 (ERROR_FILE_NOT_FOUND)
- 2026-05-12: git history 显示 commit 65f733d 已修复此问题（移除 wt.exe 路径，仅用 cmd.exe + raw_arg），但后续 commit bfd47c0 恢复了旧版 wt.exe 优先逻辑，导致回归
- 2026-05-12: 修复后 cargo test --lib 18/18 全部通过

## Eliminated

- PATH 环境变量问题（非原因，wt.exe 本身能启动但内部执行失败）
- 项目路径不存在（所有命令都失败，不只是特定项目）
- cmd.exe 回退路径有问题（代码根本没走到回退逻辑）

## Resolution

**root_cause:** `shell.rs` 的 `execute_command` 函数使用 wt.exe (Windows Terminal UWP App Execution Alias) 优先路径。从 Tauri GUI 进程 spawn wt.exe 时，`spawn()` 返回 `Ok(())`（因为 wt.exe 别名进程已启动），但 Windows Terminal 内部无法正确执行命令，报错 0x80070002 (ERROR_FILE_NOT_FOUND)。由于 `is_ok()` 检查直接返回 `Ok(())`，代码永远不会回退到 cmd.exe 路径。这是一个已知回归 —— commit 65f733d 已修复过此 bug，但后续开发重新引入了 wt.exe 优先逻辑。
**fix:** 移除 wt.exe (Windows Terminal) 优先路径，仅使用 cmd.exe 的 `start` 命令配合 `raw_arg()` 正确引用参数。添加了详细注释说明不使用 wt.exe 的原因，防止未来再次回归。
**verification:** `cargo test --lib` — 18/18 tests passed，包括 5 个 shell 参数构建测试
**files_changed:** `src-tauri/src/commands/shell.rs`
