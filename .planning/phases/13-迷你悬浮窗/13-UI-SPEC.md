### Phase 13: Execution Flash Feedback (D-17)

The flash feedback uses a green-tinted transient state:

```css
/* Flash state: triggered on command click, lasts 200ms */
.flash-active {
  background-color: oklch(0.5 0.15 155);  /* green-500 at 20% opacity overlay */
  border-color: oklch(0.5 0.15 155 / 0.4);
  transition: background-color 100ms ease-out, border-color 100ms ease-out;
}
```

Implementation: inline Tailwind classes `bg-green-500/20 border border-green-500/40` applied for 200ms via setTimeout, then removed. The transition from flash back to default uses `transition-all duration-150 ease-out`.

## Interaction Specifications

### 1. TitleBar Float Toggle Button (D-09)

#### Visual Layout

Insert a new button to the LEFT of the Settings gear button in TitleBar:

```
[Package icon "EasyPack"]  [spacer]  [PanelTop btn] [Settings btn] [Minimize btn] [Maximize btn] [Close btn]
```

#### Visual Specification

```
Button: uses existing .titlebar-button class (28px width, full height, centered icon)
Icon: PanelTop from lucide-react, 14px (w-[14px] h-[14px])
Active state (float window open): text-foreground (brighter than default)
Inactive state (float window closed): text-muted-foreground (default muted)
Tooltip: title="悬浮窗"
aria-label: "切换悬浮窗"
```

#### States

| State | Style | Notes |
|-------|-------|-------|
| Default (float hidden) | `titlebar-button` base: bg-transparent, text-muted-foreground | Matches existing TitleBar button pattern |
| Hover | `titlebar-button:hover`: bg-accent, text-foreground | Standard hover |
| Active (float visible) | `titlebar-button` + text-foreground (slightly brighter) | Visual indicator that float is open |
| Active hover | Same as inactive hover: bg-accent, text-foreground | Consistent hover behavior |

#### Interaction (D-15: toggle behavior)

| Event | Behavior |
|-------|----------|
| Click when float hidden | Creates/shows floating window |
| Click when float visible | Hides floating window |
| Click during window creation | Debounced -- no double-creation |

### 4. Command Row Component (within FloatApp)

Each command in the floating window is rendered as a compact horizontal row, NOT a grid card. This is a simplified, single-purpose variant of the main window's CommandCard.

#### Visual Specification

```
Row: h-8 flex items-center gap-2 px-2 rounded-md
     transparent bg, no border
     cursor-pointer select-none
     transition-all duration-150 ease-out

Icon: size-[14px] (14px, matching TitleBar icons)
Name: text-xs font-normal text-card-foreground
Tooltip (title attr): command string
```

#### States

| State | Background | Text | Border | Notes |
|-------|-----------|------|--------|-------|
| Default | transparent | text-card-foreground | none | Base state |
| Hover | bg-white/5 | text-card-foreground | none | Subtle lift |
| Active (mousedown) | bg-white/10 | text-card-foreground | none | Press feedback |
| Flash (post-click, D-17) | bg-green-500/20 | text-card-foreground | border-green-500/40 | 200ms green flash |
| Disabled (no project) | transparent | text-muted-foreground/40 | none | cursor-not-allowed |
| Focus (keyboard) | bg-white/5 | text-card-foreground | ring-1 ring-ring | Keyboard accessibility |

#### Flash Feedback Detail (D-17)

```typescript
// On command row click:
const [flashIndex, setFlashIndex] = useState<number | null>(null);

const handleRowClick = (index: number, command: string) => {
  if (flashIndex !== null) return; // prevent double-flash
  setFlashIndex(index);
  onExecute(command);
  setTimeout(() => setFlashIndex(null), 200);
};
```

The row gets `bg-green-500/20 border border-green-500/40` classes for 200ms. Green (not white) chosen because:
- Distinct from hover state (bg-white/5)
- Positive "executed" connotation
- Consistent with toast success coloring

#### Command Row Does NOT Show (D-16)

- No shortcut badges
- No edit/delete buttons
- No number badges
- No recording state

The floating window is a compact, read-only execution surface only.

### 6. State Synchronization (Main Window <-> Float Window)

At Claude's discretion from CONTEXT.md. Two primary options:

**Option A: Tauri Event System (recommended)**
- Main window emits `project-changed` event with `{ projectId, projectName, commands[] }`
- Float window listens for `project-changed` and re-renders
- Float window emits `command-executed` event when a button is clicked
- Main window listens for `command-executed` and runs executeCommand

**Option B: Shared Store**
- Both windows read/write to the same tauri-plugin-store
- Main window writes current project + commands on change
- Float window polls or listens for store changes
- Float window writes execute requests to store, main window picks them up

Event system (Option A) is likely better for real-time sync (FLOAT-06: "reflects changes in real-time").

## Animation Specification

Inherited from Phase 1/8/9 UI-SPEC. Phase 13 additions:

| Element | Transition | Duration | Easing |
|---------|-----------|----------|--------|
| Command row: hover | background-color fade-in | 150ms | ease-out |
| Command row: active press | background-color change | 100ms | ease-out |
| Command row: flash feedback | bg green flash | 200ms | ease-out (100ms in, 100ms out) |
| Close button: hover | bg-color + color swap | 100ms | ease (from .titlebar-button) |
| TitleBar float button: hover | bg-color fade | 100ms | ease (from .titlebar-button) |

All transitions use the standard `transition-all duration-150 ease-out` pattern established in prior phases.

The flash feedback timing (200ms) is shorter than the main window CommandCard flash (420ms) because:
- Floating window is a quick-action surface, feedback should be snappy
- User expects immediate confirmation, not a drawn-out animation
